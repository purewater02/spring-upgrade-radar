# Migration Tickets

Each finding from the Spring Upgrade Radar is now rendered as a migration ticket in the report. This makes the report actionable for teams to prioritize migration work.

## Ticket Format

Each ticket includes the following fields:

- **Title**: A descriptive title for the ticket
- **Rule**: The rule or finding that triggered the ticket
- **Why**: Explanation of why this needs attention
- **Affected files/evidence**: Files and lines that triggered the finding
- **Suggested change**: How to fix or migrate the code
- **Validation command**: Command to run to verify the fix
- **Rough effort**: Estimated effort (S/M/L/XL)

## Example Ticket

```
### Ticket 1: Spring Boot 2.x → 3.x 대형 마이그레이션

- **Rule**: spring-boot-3-5-migration
- **Why**: Spring Boot 3.5 requires Java 17+. This project is using Java 8.
- **Affected files/evidence**: 
  - `build.gradle` line 1: `javaVersion = '1.8'`
  - `pom.xml` line 1: `<java.version>1.8</java.version>`
- **Suggested change**: Change Java version to 17 or higher in build files.
- **Validation command**: `cd blog && ./mvnw test`
- **Rough effort**: XL
```

## Usage

The migration tickets are automatically included in the Markdown and HTML reports generated by the scanner.

To generate a report with migration tickets:

```bash
python3 -m spring_upgrade_radar.cli scan <project_path> \
  --target 3.5 \
  --output out/report.md \
  --html-output out/report.html \
  --tickets-json out/tickets.json \
  --jira-csv out/jira.csv \
  --github-issues-md out/github-issues.md
```

The report will include a "Migration tickets" section with all the actionable items.

## JSON, Jira CSV, and GitHub Issues Export

When `--tickets-json` is provided, the scanner also writes a structured JSON payload for downstream workflow automation.

Example JSON fields:

- `project`, `build_tool`, `spring_boot_version`, `java_version`, `target_boot`, `risk_score`
- `tickets[].id` such as `SUR-001`
- `tickets[].rule_id`, `severity`, `title`, `why`
- `tickets[].evidence`
- `tickets[].suggested_change`
- `tickets[].validation_command`
- `tickets[].rough_effort`

When `--jira-csv` is provided, the scanner writes an import-ready CSV with these columns:

- `Issue Type`
- `Summary`
- `Description`
- `Priority`
- `Labels`
- `Story Points`

Severity is mapped into Jira-friendly planning metadata: high → `Highest`/8 points, medium → `Medium`/3 points, low → `Low`/1 point.

When `--github-issues-md` is provided, the scanner writes a Markdown bundle with one section per issue:

- issue title with stable `SUR-###` ID
- GitHub labels
- issue body with why/evidence/suggested change/validation/planning sections
- starter `gh issue create` command
